### 3.1 数据库设计

#### 3.1.1 新增表结构

**1. 角色表 `role`**

```sql
CREATE TABLE role (
  id          INT PRIMARY KEY AUTO_INCREMENT COMMENT '角色ID',
  name        VARCHAR(50) NOT NULL COMMENT '角色名称，如：管理员',
  code        VARCHAR(50) UNIQUE NOT NULL COMMENT '角色标识，如：admin',
  description VARCHAR(200) COMMENT '角色描述',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT '角色表';
```

**2. 权限表 `permission`**

```sql
CREATE TABLE permission (
  id          INT PRIMARY KEY AUTO_INCREMENT COMMENT '权限ID',
  name        VARCHAR(50) NOT NULL COMMENT '权限名称，如：新增员工',
  code        VARCHAR(100) UNIQUE NOT NULL COMMENT '权限标识，如：system:emp:add',
  type        VARCHAR(20) COMMENT '类型：menu/button/api',
  parent_id   INT DEFAULT 0 COMMENT '父权限ID',
  sort        INT DEFAULT 0 COMMENT '排序',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
) COMMENT '权限表';
```

**3. 用户-角色关联表 `emp_role`**

```sql
CREATE TABLE emp_role (
  emp_id  INT NOT NULL COMMENT '员工ID',
  role_id INT NOT NULL COMMENT '角色ID',
  PRIMARY KEY (emp_id, role_id)
) COMMENT '员工-角色关联表';
```

**4. 角色-权限关联表 `role_permission`**

```sql
CREATE TABLE role_permission (
  role_id       INT NOT NULL COMMENT '角色ID',
  permission_id INT NOT NULL COMMENT '权限ID',
  PRIMARY KEY (role_id, permission_id)
) COMMENT '角色-权限关联表';
```

## 四、第二阶段：Token 刷新 + 文件上传（P1 - 重要，第 2 周）

### 4.1 Token 双 Token 机制

#### 4.1.1 方案选型

采用 **双 Token 一次性刷新** 方案：
- `accessToken`（即原 `token`）：有效期 2 小时
- `refreshToken`：有效期 7 天，用一次即失效
- 刷新时同时返回新的 `accessToken` 和新的 `refreshToken`

#### 4.1.2 数据库新增表

```sql
CREATE TABLE refresh_token (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  emp_id      INT NOT NULL COMMENT '用户ID',
  token       VARCHAR(500) UNIQUE NOT NULL COMMENT '刷新令牌',
  expire_time DATETIME NOT NULL COMMENT '过期时间',
  revoked     TINYINT DEFAULT 0 COMMENT '是否已吊销',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP
) COMMENT '刷新令牌表';
```

#### 4.1.3 新增/修改文件

| 文件 | 操作 | 说明 |
|------|------|------|
| `pojo/LoginInfo.java` | 修改 | 增加 `refreshToken` + `expiresIn` |
| `controller/AuthController.java` | 新增 | 刷新 Token 接口 `POST /auth/refresh` |
| `service/AuthService.java` | 新增 | Token 刷新业务逻辑 |
| `mapper/RefreshTokenMapper.java` | 新增 | 刷新令牌数据访问 |
| `utils/JwtUtils.java` | 修改 | 区分生成 accessToken / refreshToken |
| `interceptor/TokenInterceptor.java` | 修改 | 明确 401 错误消息区分 |

#### 4.1.4 Token 错误码规范

| 场景 | HTTP 状态码 | 响应 msg |
|------|-------------|----------|
| Token 为空 | 401 | `token不能为空` |
| Token 无效/过期 | 401 | `token已过期` |
| RefreshToken 无效/失效 | 401 | `refreshToken已失效` |
| 权限不足 | 403 | `权限不足，无法访问` |

### 4.2 文件上传接口规范化

#### 4.2.1 现状问题

- 当前路径：`POST /upload` → 需调整为 `POST /files/upload`
- 响应格式：直接返回 URL 字符串 → 需返回结构化对象
- 缺少：文件分类、文件删除、类型校验、大小校验

#### 4.2.2 改造方案

| 项 | 改造前 | 改造后 |
|----|--------|--------|
| 上传接口 | `POST /upload` | `POST /files/upload` |
| 请求参数 | `file` | `file` + `type`（可选） |
| 响应 | `data: "url"` | `data: { url, name, size }` |
| 删除接口 | 无 | `DELETE /files?url=xxx` |
| 类型校验 | 无 | jpg/png/gif/pdf/doc/docx/xls/xlsx |
| 大小限制 | 全局 10MB | 图片 ≤ 5MB，文档 ≤ 20MB |

#### 4.2.3 新增/修改文件

| 文件 | 操作 |
|------|------|
| `controller/UploadController.java` | 修改路径、参数、响应结构 |

## 六、技术风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 权限注解 AOP 切面与日志切面顺序问题 | 中 | 中 | 使用 `@Order` 明确执行顺序：权限校验 → 操作日志 |
| RefreshToken 并发刷新 | 低 | 低 | 加分布式锁或数据库乐观锁，防止并发下重复使用 |
| OSS 删除权限配置 | 中 | 高 | 提前确认 RAM 账号是否有 DeleteObject 权限 |
| 前端接口路径变更兼容 | 高 | 中 | 旧路径 `/upload` 保留一段时间做兼容，或前端同步改 |
| 权限数据缓存 | 中 | 中 | 权限查询较频繁，可考虑加入 Redis 缓存（二期优化） |
